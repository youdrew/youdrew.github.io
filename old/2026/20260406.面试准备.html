Q2: 编辑器的属性编辑器（Inspector）你是怎么设计和实现的？

参考答案：

属性编辑器的核心是 反射 + 数据绑定：

  1. 反射系统：通过类型注册机制（或宏/代码生成），将类的字段信息（类型、名称、Range、Tooltip 等元数据）注册到反射系统中
  2. 属性绑定：Inspector 根据反射信息，为每个字段动态创建对应的 UI 控件（int → Slider、float → DragFloat、enum → ComboBox、Object → ObjectPicker 等）
  3. 数据流向：UI 控件 ↔ 属性绑定层 ↔ 对象实例，双向同步
  4. Undo/Redo：属性修改前记录旧值到 Undo 栈，支持回滚

Q3: 蓝图/节点编辑器的核心架构是什么？如何处理节点间的数据流？

参考答案：

蓝图编辑器 = 节点图数据结构 + 图编辑 UI + 编译执行引擎

  1. 数据模型：

    • GraphNode：包含输入/输出 Pin，每个 Pin 有类型（Exec/数据类型）
    • GraphEdge：连接 Output Pin → Input Pin
    • Graph：管理所有 Node 和 Edge

  2. 图编辑 UI：

    • Canvas 画布：平移、缩放、框选
    • 节点渲染：标题栏、Pin 端口、连线（贝塞尔曲线）
    • 交互：拖拽连线、右键菜单创建节点、断开连接

  3. 编译执行：

    • 解释执行：遍历图节点，从 Root/Event 节点开始 DFS 执行
    • 编译执行：将节点图翻译为字节码或 C++/Lua 代码，性能更高
    • 类型检查：编译期验证 Pin 连接的类型兼容性

Q4: PCG（程序化内容生成）系统的架构设计思路？

参考答案：

PCG 系统通常采用 Graph-Based Pipeline：

  1. PCG Graph：节点图，每个节点是一个 Generator 或 Filter

    • Generator：地形生成（Perlin Noise）、植被散布、道路生成等
    • Filter：密度过滤、高度约束、碰撞检测等
    • Transform：旋转、缩放、偏移

  2. 分块调度（Tile-based）：

    • 世界划分成 Tile，按需加载生成
    • 每个 Tile 独立计算，支持并行
    • Tile 间需要处理接缝（Seam Stitching）

  3. LOD 支持：远处 Tile 简化生成，近处精细生成

  4. 编辑器集成：

    • 节点编辑器可视化 PCG Graph
    • 实时预览：参数修改后增量重新生成
    • 手动覆盖层（Override）：允许美术在 PCG 基础上手动调整

B. 引擎基础

Q6: 场景管理的常见方案有哪些？如何处理大规模场景？

参考答案：

  1. 场景图（Scene Graph）：树形结构，适合小型场景，Transform 层级传递
  2. ECS 架构：Entity-Component-System，数据导向，缓存友好，适合大规模实体
  3. 空间划分：
    • QuadTree / Octree：用于视锥剔除、碰撞检测
    • BVH（Bounding Volume Hierarchy）：渲染加速
    • Grid / Virtual Texture：大世界分块

大世界方案：

  • World Partition（UE5 风格）：空间划分为 Grid Cell，按需加载/卸载
  • Streaming Level：分块流式加载，LOD 切换
  • HLOD：远处多个 Static Mesh 合并为一个，减少 Draw Call

Q7: ECS 架构是什么？和传统 OOP 有什么区别？

参考答案：

OOP ECS
数据布局 对象分散在堆上 Component 连续内存排列
缓存友好性 差（指针跳转） 好（数据局部性）
逻辑组织 方法绑定在类上 System 独立遍历 Component
组合性 需要多重继承/组合模式 天然组合（挂 Component 即可）

核心概念：

  • Entity：ID 标识，不存数据
  • Component：纯数据（Position, Velocity, Health…）
  • System：纯逻辑（MovementSystem 遍历所有 Position+Velocity 组件）

典型应用： Unity DOTS、EnTT（C++）、Flecs

Q9: RAG 是什么？你是怎么实现的？

参考答案：

RAG（Retrieval-Augmented Generation） = 先检索相关文档，再让 LLM 基于检索结果回答。

实现流程：

优化点：

  • Chunk 策略：按语义切分（Sentence / Paragraph），Overlap 滑窗
  • 混合检索：向量检索 + 关键词检索（BM25）融合
  • Reranker：用 Cross-Encoder 对 Top-K 结果重排序
  • Metadata 过滤：按文档类型/时间等元数据过滤

Q10: Function Calling 的原理是什么？你在项目中怎么用的？

参考答案：

原理： LLM 不直接执行函数，而是 输出结构化的函数调用意图（函数名 + 参数），由宿主程序解析并执行。

流程：

在编辑器中的应用：

  • 用户说”创建一个 Cube 放在 (0,1,0)” → LLM 调用 spawn_actor(type="Cube", position=[0,1,0])
  • “把选中物体的材质换成红色” → LLM 调用 set_material(selection, color="red")
  • 多步操作链：LLM 可以连续调用多个函数完成复杂任务

Q11: ComfyUI 的工作流原理是什么？

参考答案：

ComfyUI 是 Stable Diffusion 的 节点式图形化工作流框架：

  1. 节点图（Graph）：每个节点是一个处理单元（模型加载、Prompt、Sampler、VAE Decode、Save Image 等）
  2. 数据流：节点间通过连接传递 Tensor/数据
  3. 执行引擎：拓扑排序后按序执行，支持缓存中间结果
  4. 优势：
    • 可视化编排，非程序员也能搭建复杂工作流
    • 节点可复用、可组合
    • 方便实验不同参数组合
    • 支持自定义节点扩展

与编辑器的关联： ComfyUI 的节点图思想和蓝图编辑器本质相同——都是 数据驱动的 DAG 执行模型。

Q12: ControlNet 是什么？它解决了什么问题？

参考答案：

ControlNet 为预训练 Diffusion Model 添加 空间控制能力。

解决的问题： 原始 Text-to-Image 无法精确控制生成图像的构图、姿态、边缘等空间结构。

原理：

  • 在 Stable Diffusion 的 U-Net 每个 Block 上添加一个并行的 “Zero Convolution” 分支
  • 输入额外的条件图（边缘检测/Canny、深度图、姿态/OpenPose、法线贴图等）
  • 训练时只训练 ControlNet 分支，冻结原始模型

常见 ControlNet 类型：

  • Canny Edge：控制轮廓
  • Depth：控制深度结构
  • OpenPose：控制人物姿态
  • Segment：控制区域分割
  • Scribble：控制草图

Q13: LoRA 是什么？为什么它很受欢迎？

参考答案：

LoRA（Low-Rank Adaptation） 是一种参数高效微调方法。

核心思想： 不修改原始模型权重 W，而是添加一个低秩分解矩阵 ΔW = A × B，其中 A ∈ R^(d×r)，B ∈ R^(r×d)，r << d。

优势：

  • 参数量小：通常只训练原始模型 0.1%-1% 的参数
  • 文件小：几 MB 到几百 MB（vs 全量微调几 GB）
  • 可组合：多个 LoRA 可以叠加使用
  • 切换快：运行时加载/卸载 LoRA 权重，无需多个模型副本

在 AIGC 中的应用：

  • 风格 LoRA：训练特定美术风格（如水墨画、赛博朋克）
  • 角色 LoRA：训练特定角色外观
  • 概念 LoRA：训练特定物体/服装

三、技术深度追问准备

Q17: 如何优化编辑器在大数据量场景下的性能？

  1. 视锥剔除（Frustum Culling）：只渲染可见物体
  2. LOD 系统：远处用简化模型
  3. Instancing：相同 Mesh 合批渲染（GPU Instancing）
  4. 虚拟化列表：Hierarchy/Inspector 等面板使用虚拟滚动，只渲染可见行
  5. 异步加载：资源按需加载，不阻塞主线程
  6. 延迟序列化：只在保存时序列化，运行时用指针引用
  7. 空间索引：Octree/BVH 加速拾取和碰撞检测

R2: 渲染队列（Render Queue）是什么？不同队列的顺序是怎样的？

参考答案：

渲染队列决定了物体的绘制顺序，合理排序可以减少状态切换和 Overdraw。

UE/Unity 常用队列顺序（从前到后）：

队列 渲染顺序 适用物体 说明
Background 0–999 天空盒等背景 最先渲染
Geometry 1000–1999 不透明物体 按距离排序，最优效率
AlphaTest 2000–2450 Alpha Test 物体（树叶、草） 需要 MRT 时
Transparent 2460–2999 半透明物体 从远到近排序，背面→正面
Overlay 3000–4000 特效、UI 最后渲染

关键点：

  • 不透明物体：从近到远排序（减少 Overdraw，因为后面的会被深度测试丢弃）
  • 半透明物体：必须从远到近排序（因为需要 Blending 混合，且无法深度写入）
  • Early-Z / Z-Prepass：先用简单 Shader 写入深度，再渲染不透明物体（利用硬件 Early-Z）

R3: 什么是 Draw Call？Batch？它们的区别和优化方法？

参考答案：

Draw Call：CPU 向 GPU 发起的渲染调用命令（DrawElements / DrawInstanced 等），每次 Draw Call 都有 CPU 开销（状态检查、命令绑定）。

Batch（合批）：将多个 Draw Call 合并为一次，减少 CPU 开销。

类型 原理 限制
Static Batching 静态物体合并为同一 Mesh，运行时不能移动/旋转/缩放 增加内存，换材质合批失效
Dynamic Batching 运行时动态合批小型网格（顶点数 < 900） 只适合小物体，CPU 计算合批开销
GPU Instancing 一次 Draw Call 渲染 N 个相同 Mesh、不同 Transform 的实例 适合大量重复物体（树木、子弹、特效粒子）
SRP Batcher 相同 Shader 的材质合批（仅变换矩阵/颜色等 CB 变化） 适合大量小物体的 UI、骨骼动画

Draw Call 优化的核心思路：

  1. 合批：减少 Draw Call 数量
  2. 减少状态切换：按 Shader / 材质分组渲染，减少 SetShader / SetTexture 切换
  3. GPU Instancing：大量重复物体用 Instancing
  4. 遮挡剔除（Occlusion Culling）：不渲染被遮挡的物体，减少无效 Draw Call

  • Forward+ / Clustered Forward：结合两者优点，兼顾透明物体和大量光源

R5: PBR（基于物理的渲染）的基本原理？Metallic/Roughness 工作流？

参考答案：

PBR 核心思想： 用物理公式替代经验性的光照模型，让美术资源（贴图）在不同光照下都能自然真实。

两个主要工作流：

工作流 使用贴图 适用场景
Metallic/Roughness（M/R） Albedo + Metallic + Roughness + AO 工业/科幻/写实，美术更直观
Specular/Glossiness（S/G） Diffuse + Specular + Glossiness + AO 传统工作流，可做更多艺术效果

Metallic/Roughness 贴图含义：

  • Albedo：无光反照率贴图（物体本色，黑色=完全吸光）
  • Metallic：金属度贴图（0=非金属，1=纯金属）
  • Roughness：粗糙度贴图（0=完全光滑/镜面反射，1=完全粗糙/漫反射）
  • Normal：法线贴图（模拟表面微观凹凸）
  • AO（Ambient Occlusion）：环境光遮蔽贴图（缝隙/角落变暗）

R6: 什么是光照贴图（Lightmap）？烘焙光照和实时光照的区别？

参考答案：

光照贴图（Lightmap）： 将静态光源的光照信息预先计算并存储到一张 2D 贴图中，运行时采样使用。

烘焙光照 vs 实时光照：

烘焙光照（Baked） 实时光照（Realtime） 混合光照（Mixed）
计算时机 离线烘焙时 运行时实时计算 部分烘焙+部分实时
阴影质量 高（可多次弹射、全局光照） 受限于ShadowMap分辨率 中等
运行时开销 低（只有采样开销） 高（每帧计算） 中等
适用场景 静态场景光照 动态物体/光源 大世界（UE5 Lumen）
移动端 推荐 尽量避免 谨慎使用

光照贴图技术要点：

  • UV Layout：静态物体需要第二套 UV（确保无岛接缝、面积比例适中）
  • Atlas packing：多个物体共享一张光照图集（减少 Draw Call）
  • Directional Lightmap：额外存储法线方向信息，支持法线贴图在烘焙光照下正确显示
  • 实例化渲染（UE Instance）：相同 Mesh 共享光照图采样，但 UV 不同（GPU Instancing）

新一代技术： UE5 Lumen、Unity HDRP Ray Tracing / Path Tracing — 用硬件/软件光线追踪实现实时全局光照

R7: 渲染器中如何处理半透明物体的排序和渲染？

参考答案：

半透明物体是渲染中最复杂的问题之一，因为需要 Blending 混合且不能写入深度。

核心问题： 错误的排序会导致半透明物体内部相互穿插、遮挡错误。

UE/Unity 的处理策略：

Order-Independent Transparency（OIPT）高级方案：

  • Weighted Blended OIT：按权重混合，不需要排序
  • Depth Peeling：多层深度剥离，渲染多层半透明
  • Adaptive Transparency（AT）： UE5 Nanite 使用的透明方案

R10: 什么是 GPU 虚拟纹理（Virtual Texture）？它解决了什么问题？

参考答案：

问题背景： 超大贴图（如 8K×8K）无法一次性全部加载到显存，且 Draw Call 开销巨大。

GPU 虚拟纹理（VT）核心思想： 像操作系统的虚拟内存一样，按需从磁盘/内存流式加载纹理分页（Page）。

实现原理：

UE5 Virtual Shadow Map： 整个场景用一个巨大的虚拟阴影图（8192×8192），每个物体按需占用一小块，按需流式加载

优势：

  • 解决显存瓶颈：不需要一次性加载全部贴图
  • 减少 Draw Call：相同 Virtual Tile 的物体可以合批
  • 支持超大贴图（8K+）

挑战：

  • Page Table 查询开销
  • 流式加载延迟（需要 Mipmap Chain 降级过渡）
  • 纹理过滤在跨 Tile 时需要特殊处理

R12: 渲染器架构中的 RHI（Rendering Hardware Interface）是什么？

参考答案：

RHI 是渲染器的跨平台抽象层，将上层渲染逻辑与底层图形 API（DX11/DX12/Vulkan/Metal/OpenGL）解耦。

典型 RHI 架构：

RHI 核心类型：

  • FRHICommandList：跨平台渲染指令列表
  • FRHIUniformBuffer：跨平台 Uniform Buffer（UBO/Constant Buffer）
  • FRHITexture：跨平台纹理句柄
  • FRHIVertexDeclaration：顶点格式声明
  • FRHIPipelineState：Pipeline State Object（PSO）

设计意义：

  • 一次编写，多平台运行
  • 引擎层不需要感知 DX/Vulkan/Metal 差异
  • 方便做 API Feature Level 的兼容处理

R13: 渲染器如何处理渲染线程和游戏线程的同步？

参考答案：

渲染管线需要多线程并行处理以提升性能：

线程模型（以 UE 为例）：

关键同步点：

  1. Game Thread → Render Thread：用 Queue（FIFO）传递渲染命令
    • UE：ENQUEUE_RENDER_COMMAND / FlushRenderingCommands()
  2. Render Thread → GPU：同步点（Sync），等待 GPU 完成上一帧
    • RHIFence：GPU 完成信号量
  3. GPU → Game Thread：读取 GPU 结果（如 GPU Particle 查询）
    • 需要 Stall（CPU 等待 GPU），尽量避免

多线程渲染优化：

  • GPU Drivern Rendering：CPU 端由 GPU 自己驱动 Draw Call 生成，减少 CPU 开销
  • Parallel Rendering：将渲染指令列表拆分并行构建（UE 的 Parallel Command List）
  • Async Compute：GPU 内部并行执行图形和计算指令

R14: 什么是 SRP（Scriptable Render Pipeline）？URP 和 HDRP 的区别？

参考答案：

SRP（可编程渲染管线） 是 Unity 2019 引入的渲染架构，允许用 C# 代码完全控制渲染流程。

对比 Built-in 渲染管线：

Unity URP（Universal Render Pipeline）：

  • 定位：通用渲染，兼顾性能和质量
  • 目标平台：移动端、PC、主机（中低端硬件）
  • 特点：Forward Renderer，轻量，支持 Instancing、SRP Batcher
  • Shader 特色：Shader Graph 友好，大量 PBR 节点
  • 限制：不支持实时光线追踪，透明排序较简化

Unity HDRP（High Definition Render Pipeline）：

  • 定位：高端渲染，影视级质量
  • 目标平台：高端 PC、主机（PS5/Xbox Series X）
  • 特点：
    • Deferred Renderer（默认）
    • 物理相机（Physical Camera，焦距/光圈/散景）
    • 体积雾/体积光
    • Ray Tracing / Path Tracing（反射、阴影、AO）
    • Subsurface Scattering（皮肤/树叶透光）
  • 要求：高端硬件，移动端不支持
