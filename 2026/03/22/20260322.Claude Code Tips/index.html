Claude Code 使用技巧整理

这篇文档主要记录我在使用 Claude Code 过程中的一些稳定工作流、提问方式和插件使用经验。目标不是把所有功能列一遍，而是整理出真正能提高交付效率的做法。

Claude Code 适合做什么

如果从游戏开发、工具链开发和内容工程的视角来看，Claude Code 更适合以下几类任务：

  • 理解已有工程结构
  • 批量重构脚本或工具代码
  • 补测试、补文档、补脚手架
  • 对接第三方 SDK、服务接口或命令行工具
  • 做中等复杂度的多文件改动

它的优势不是“直接无脑一把梭”，而是：

  • 长上下文理解能力比较稳定
  • 对代码解释、重构、归纳总结表现较好
  • 适合先分析、再执行的工作流

如果任务非常碎、非常即时，比如一边写一边需要行级补全，那还是 IDE 内补全工具更直接。

Plugin 使用总结

插件或者外部工具接入，真正的价值不是“功能变多”，而是让 Claude Code 可以从“只会说”变成“能查、能跑、能验证”。

我的判断标准通常是三条：

  1. 能不能减少手动切换上下文
  2. 能不能把分析结果落到真实工程上
  3. 能不能形成稳定工作流，而不是偶尔演示一次

ralph-loop （循环插件）

ralph-loop 这类插件我更倾向把它看成“循环执行框架”或者“任务闭环增强器”。

它比较适合下面这些场景：

  • 一个任务需要多轮分析、执行、检查
  • 需要让 AI 按固定节奏反复迭代
  • 需要把大任务拆成可观察的小回合

自动化同意问题

如果你的目标只是“允许它直接修改当前工作区里的文件，不要每次都弹确认”，优先用 Claude Code 自带的权限模式，而不是指望插件本身绕过确认。

我建议分两档理解：

方案 1：只自动同意编辑文件

这是更稳的做法。

Claude Code 有一个 acceptEdits 模式，核心效果是：

  • 工作区内的文件编辑可以批量接受
  • 但执行命令、联网请求、其他有副作用的操作，仍然会继续询问

如果你主要烦的是“改这个文件能不能同意”这种提示，那应该优先用这一档。

可以通过 /config 或 /permissions 检查当前模式，也可以在设置文件里显式写上：

常见放置位置：

  • 全局生效：~/.claude/settings.json
  • 当前仓库本地生效：.claude/settings.local.json
  • 团队共享配置：.claude/settings.json

如果你只是想在自己机器上对当前项目放开，最适合放到 .claude/settings.local.json。这样不会提交进仓库，也不会影响别的项目。

方案 2：连命令确认也一起跳过

这个模式相当于直接跳过权限确认，适合你完全信任当前仓库、并且明确知道它可能会自动执行改文件、跑命令、调用工具等操作的场景。

但这一档风险明显更高，不建议默认长期打开。更适合：

  • 自己的个人仓库
  • 隔离良好的测试环境
  • 临时做高频迭代任务

不太建议直接用于：

  • 来路不明的仓库
  • 混有敏感配置的工程
  • 会执行部署、发布、数据库操作的项目

实战建议

如果你的诉求只是“ralph-loop 在多轮执行时别一直问我能不能改文件”，那优先级应该是：

  1. 先把默认权限模式切到 acceptEdits
  2. 用 .claude/settings.local.json 只对当前仓库生效
  3. 只有在你连命令确认都嫌打断流程时，才改用 --dangerously-skip-permissions

简单说：

  • 只想免掉文件编辑确认，用 acceptEdits
  • 想把所有确认基本都跳过，用 --dangerously-skip-permissions

前者适合日常主力使用，后者适合你明确接受风险时再开。

简单说，ralph-loop 不是让 Claude Code “更聪明”，而是让你的任务编排更稳定。

Oh My Claude Code（OMC，Claude智能体集合）

主打一个零成本学习Claude的一个工具。

提供了5种模式：
autopilot：全自动模式，整合了ralph（持久性），ultrawork（并行），plan（战略思维）整合在了一起。

autopilot: 做一点什么。

ultrapilot：更激进的自动化（autopilot的升级版本，最多5个并行的autopilot）

🔸 Autopilot（完全自主） - 从规划、实施到测试的全流程自动化

🔸 Ultra Pilot（并行加速） - 最多5个并行工作者同时处理，速度提升5倍

🔸 Swarm（协作团队） - 多个智能体像开发团队一样协作，从共享任务池领取工作

🔸 Pipeline（流水线） - 按固定顺序串联智能体，适合必须按步骤推进的工作流

🔸 EcoCode（经济模式） - 在保持效率的前提下最大化节省 token 消耗

● 这是 OMC 状态栏的显示，各项含义：

  • [OMC#4.11.2] — OMC 插件版本号
  • session:33m — 当前会话已持续 33 分钟
  • ctx:32% — 上下文窗口已使用 32%
  • T:17 — Tool calls，本次会话已执行的工具新调用次数（17 次）
  • A:1 — Agents，当前活跃的子代理数量（1 个）

文档检索类插件

如果插件可以帮助 Claude Code 直接读取官方文档、接口说明或者本地知识库，那么它在接第三方 SDK、引擎 API、云服务时非常有用。

适合场景：

  • 查 API 参数
  • 对照版本差异
  • 核对配置项含义
  • 减少“模型凭印象回答”

这类插件的核心价值是降低幻觉率。

仓库与 Issue 协同类插件

如果你的工作流经常围绕 Git、Issue、PR 进行，那么能直接读仓库状态、变更记录、评审意见的插件会非常实用。

适合场景：

  • 根据 Issue 修 bug
  • 结合 PR review comment 改代码
  • 总结当前分支改动
  • 生成提交说明或变更摘要

这类插件最适合团队协作环境，而不是单人写 demo。

浏览器自动化或页面验证类插件

做前端、后台管理系统或者文档站点时，能操作页面、检查 DOM、截图、回归验证的插件也很有价值。

适合场景：

  • 检查页面是否正常渲染
  • 验证按钮点击流程
  • 截图留档
  • 复现前端交互 bug

如果你的任务需要“改完后马上验证页面结果”，这类插件会比纯文本分析更有效。

Claude Code 真正好用的地方，不是一次生成很多代码，而是它可以参与完整的软件交付链路：

  • 先读工程
  • 再理解约束
  • 然后分步实施
  • 最后补验证和总结

如果再配合合适的插件，尤其是像 ralph-loop 这种帮助任务闭环和节奏控制的工具，整体体验会比“只聊天、不落地”的 AI 协作模式更强。
