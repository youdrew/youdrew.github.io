Anthropic 还发了一篇架构长文,讲 Managed Agents 为什么要这么设计。简单说,目标是做一个能扛住”未来还没想到的程序”的通用框架,思路跟操作系统当年把硬件虚拟化成 process、file 一样,read() 不需要知道底层是 1970 年的磁盘包还是现代 SSD。Managed Agents 把 agent 拆成三件套:session 是追加写的事件日志,harness 是调用 Claude 并路由工具调用的循环,sandbox 是真正跑代码改文件的环境,三者接口固定、各自可换。最早塞同一个 container,结果这个 container 变成”宠物”——坏了就丢状态、卡住要手动修、debug 还得开 shell,经常连带用户数据,而且客户想把 Claude 接进自家 VPC(虚拟私有云,就是企业自己的内网)要么 peer 网络要么把 harness 跑过去。解法是把”大脑”Claude 加 harness 从 container 拿出来,调 sandbox 跟调其他工具一样,就是 execute(name, input) 返回一段字符串;sandbox 死了就是工具调用失败,harness 把失败回传给 Claude 让它决定要不要重试,新 sandbox 用 provision 起一份就行。harness 自己也是可替换的,session log 在外面持久化,任何 harness 都能用 wake 和 getSession 从最后一个事件恢复。安全上以前 token 和代码都在一个 container 里,prompt injection(在提示词里埋陷阱骗模型)只要骗 Claude 读自己环境就能拿到凭据,现在 token 通过 Git remote、OAuth vault(把登录凭据锁在外面的保险箱)、MCP(可以理解为模型和外部工具之间的通用插头)proxy 从沙箱外注入,harness 自己都不知道凭据是什么。最后一个关键点:session 不是 Claude 的 context window(上下文窗口,就是模型一次能”看见”的文字总量),长期任务经常超 context,Anthropic 之前讨论过压缩、记忆、裁剪这些方法,都是对 token 做不可逆取舍;Managed Agents 把 session 当成”活在 context 之外的 context 对象”,提供 getEvents 让大脑按位置切片去查”刚才发生了什么”,harness 再决定怎么塞回 Claude 的上下文。效果是 p50(中位数)首 token 时间下降约 60%,p95(慢的那一头)下降超过 90%,而且一个脑子可以接多只手,一只手也能在多个脑子之间传。喜欢这种把复杂系统讲清楚的工程师朋友,这篇值得收藏慢慢读。

Anthropic 的产品博客同步更新了一条 Managed Agents 的新能力:self-hosted sandboxes 公测加 MCP tunnels 研究预览,核心就是把执行边界拉回企业自己的网络里。自托管沙箱这块,Anthropic 跑 agent loop、上下文管理、错误恢复,工具执行跑在你自己基础设施或托管 provider 上,敏感文件、仓库、依赖、凭据不出你的边界。可选的 sandbox provider 包括 Cloudflare(用 microVM + isolates,零信任 secrets 注入、自定义 egress proxy)、Daytona(完整可组合计算机、长时间有状态、SSH / 预览 URL 可访问)、Modal(亚秒冷启动、CPU/GPU 弹性)、Vercel Sandbox(VM + VPC peering + BYOC(自带云),毫秒级启动、凭据在网络边界注入);已上车的客户有 Amplitude 在 Managed Agents 加 Cloudflare 上做内部 Design Agent,Clay 的 GTM agent Sculptor 跑在 Daytona,Rogo 的金融分析师 agent 跑在 Vercel Sandbox。MCP tunnels 这块,让 agent 能调到内网的 MCP server,内部数据库、私有 API、知识库、工单系统都不用开公网,你部署一个轻量 gateway,只做单向 outbound,traffic 端到端加密,Managed Agents 和 Messages API 都支持,在 Claude Console 的 workspace 设置里管。说白了就是,以前 Claude 的脑子和手脚都跑在 Anthropic 那边,现在允许企业把手脚留在自己内网里、让脑子通过 MCP tunnels 召唤进来,对合规和数据敏感的公司是大利好。

这是一篇架构层面的长文,讲 Managed Agents 为什么这么设计:目标是做出一个能扛住”未来还没想到的程序”的 meta-harness。操作系统的解法是把硬件虚拟化成 process、file 这类足够通用的抽象,read() 不知道也不在乎底层是 1970 年的磁盘包还是现代 SSD。Managed Agents 同理,把 agent 拆成三件套——

  • session:追加写的事件日志
  • harness:调用 Claude、路由工具调用的循环
  • sandbox:真正跑代码、改文件的环境

三者各自独立可替换,接口固定、实现随便换。

最早他们把所有东西塞进同一个 container,代价是这个 container 变成了 “pet”——坏了就丢状态、卡住就要手动修、debug 还得开 shell 进去,常常连带用户数据。第二个问题是 harness 默认所有资源都在它身边,所以客户想把 Claude 接进自家 VPC,要么和我们 peer 网络,要么把 harness 跑在他们的环境里。

解法是把”大脑”(Claude + harness)从 container 里拿出来:它现在调 sandbox 的方式跟调别的工具一样,就是 execute(name, input) → string。Sandbox 死了就是工具调用失败,harness 把失败回传,Claude 决定要不要重试,新 sandbox 用 provision({resources}) 起一份就行。Harness 自己也是 cattle——session log 在外面持久化,crash 后任何 harness 都能用 wake(sessionId) + getSession(id) 从最后一个事件恢复。

安全边界上,以前 token 和生成的代码都在一个 container 里,prompt injection 只要骗 Claude 读自己环境就能拿到 credential。现在 token 通过 Git remote、OAuth vault、MCP proxy 等方式从沙箱外注入,harness 自己都不知道凭据是什么。

最后还有一个很关键的点:session 不是 Claude 的 context window。Long-horizon 任务经常会超出 context,Anthropic 在之前的 context engineering 里讨论过 compaction、memory、trimming,但这些做法都是对 token 做不可逆的取舍。Managed Agents 把 session 当成”活在 context 之外的 context 对象”,提供 getEvents() 让大脑按位置切片去问”我刚才那段发生了什么”,harness 再决定怎么塞回 Claude 的 context window。这样存储的归存储、context 管理的归 harness,两件事解耦。

效果:p50 TTFT 下降约 60%,p95 下降超过 90%。而且现在一个 brain 可以接多个 hand,hand 也可以在多个 brain 之间传递。

“We virtualized the components of an agent: a session, a harness, and a sandbox. This allows the implementation of each to be swapped without disturbing the others.”

Anthropic与AE Studio联合提出梯度路由辅助模块（GRAM）方法，通过在Transformer每层添加可移除的神经元模块，使模型在训练时将病毒学、网络安全、核物理、专业编程语言等双重用途知识仅路由到对应模块，而非扩散至全局。训练后删除模块即可消除该能力，保留则供可信部署使用。实验在合成数据、真实数据及50M到5B参数模型上测试，GRAM效果与数据过滤相当，移除模块不降低通用性能，且比事后“遗忘”技术更难恢复。该研究为平衡双重用途知识的安全访问与有益使用提供了更鲁棒的方案。
